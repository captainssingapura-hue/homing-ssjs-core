# Release 0.5.4 — Bring Your Own Tree, Typed

A point release on top of 0.5.3. Where 0.5.3 opened the door to *bringing your
own tree* — `DocTreeSource` + `RigidDoc.fromNode(...)` let a Doc hand the
endpoints a pre-built tree — 0.5.4 makes that path **first-class and typed**, and
fixes the one thing that made the 0.5.3 door awkward to walk through: the fragile
child-index path.

The headline is **`RigidDocV2`**: a name-path doc assembled by a *bring your own
tree* adapter, where the tree's shape is enforced by the node type itself rather
than checked after the fact. It lives **alongside** V1 — nothing existing changes.

## What's in the box

### `RigidDocV2` — a name-path doc from parent-pointer nodes

The successor to `RigidDoc`. You describe your hierarchy as a flat list of
**parent-pointer nodes** and hand it, with one content provider, to
`RigidDocV2.fromNodes(...)`. The adapter inverts the pointers, validates, and
serves a `DocTreeV2`.

```java
RigidNode<Src> animals = RigidNode.root(a, new NodeName("animals"));       // title -> "Animals"
RigidNode<Src> turtle  = animals.child(t, new NodeName("turtle"));         // level pinned to parent + 1

RigidDocV2 doc = RigidDocV2.fromNodes(
        uuid, "Animals", summary, "GUIDE",
        () -> List.of(animals, turtle),   // the nodes, built lazily
        this::contentFor);                // the one provider: Src -> RigidNodeContent
```

### `RigidNode<T>` — the tree shape *is* the type

Each node wraps one of **your** source nodes (`T`), holds an object reference to
its parent (a root's is `null`), and stamps its `level`. This representation makes
the expensive malformations **unrepresentable**, so the normalizer never has to
check them:

- **multiple parents** — a node has one `parent` field;
- **cycles** — the node is immutable and its parent must exist first, so
  construction is inherently parent-first; you can't tie a loop;
- **a wrong level** — pinned to `parent.level + 1` at construction time.

Build top-down with `RigidNode.root(source, name[, title])` then
`parent.child(source, name[, title])`, exactly as a DFS descends. There are no
ids — the object *is* the identity, the parent reference *is* the edge. The
`title` is optional: omit it and it defaults to the humanized name
(`"dancing-animals"` → `"Dancing Animals"`, via `NodeName.defaultTitle()`).

### One content provider, not per-node bodies

Content is **not** stored on a node. A single `Function<T, RigidNodeContent>`,
handed to the adapter, resolves every node's body from its `source` — free to
dispatch however the downstream likes (leaf vs branch, by kind, …). Structure
(the nodes) and content (the one function) stay apart, with no per-node content
thunks.

### Name-path identity — stable across reordering

Content is addressed by a node's **name-path** — the `/`-joined chain of
`NodeName`s from the root (`"animals/turtle"`) — not a child-index like `1/0`.
Insert or reorder siblings and the address is unchanged, so anchors, deep-links,
and the content lookup all survive edits.

- **`NodeName`** (`…composed.text`) — a URL-safe, sibling-unique id, capped at 48
  chars (`[A-Za-z0-9._-]`, `/` reserved as the separator). `NodeName.slug(label)`
  derives one from an arbitrary label; `defaultTitle()` humanizes it back.
- **`NodeKey`** — a new entry in the substrate's closed `DimensionKey` vocabulary
  (wire tag `nodeName`). Every V2 structure node carries it, so the client rebuilds
  the same name-path while walking — no stamped ids on the wire.

### Validation — only what the shape can't guarantee

`RigidNodeNormalizer` assembles the flat node list in one pass that both **builds**
the `DocTreeV2` and validates it. Since the type already rules out cycles,
multi-parent, and wrong levels, only three things remain, each a named
`MalformedTreeException`:

- **not exactly one root** (one `null`-parent node);
- a node **unreachable** from the root (its parent is outside the provided list);
- two **siblings sharing a name** (an ambiguous name-path).

### Wire + client

- **`DocTreeV2` / `DocTreeV2JsonWriter`** — the same two-part
  `{structure, content}` payload as V1, with content keyed by **name-path** instead
  of child-index. The leaf/segment value shape is byte-for-byte the V1 shape (the
  writer reuses V1's leaf serializer); only the key differs.
- **`DocTreeV2Source`** — a Doc that supplies its own `DocTreeV2`. The existing
  `/doc-tree` and `/doc-tree-content` endpoints recognise it and serve the V2
  payload (resource-backed segments are addressed by name-path), taking precedence
  over the index-path `DocTreeSource`.
- **Client** — `renderDocTree` **feature-detects** the `nodeName` dimension: when
  present it keys content by the name-chain and gives each section a stable,
  name-based DOM id / anchor (`doc-node-animals_turtle`); when absent it falls back
  to today's child-index path. No flag, and V1 rendering is untouched.

## Breaking changes

**None.** This release is purely additive:

- V1 `RigidDoc`, `DocNode`, `DocTreeSource`, `RigidDoc.fromNode(...)`, the
  index-path `DocTree`, and the existing wire format are all unchanged.
- A V1 doc (no `nodeName` dimension) serialises and renders exactly as before; the
  client falls back to index-path keying for it.

**One source-compat note:** the sealed `DimensionKey` gains a permitted type
(`NodeKey`). Downstream code that reads dimensions generically (by tag, as the
substrate intends) is unaffected. Only an *exhaustive* `switch` over `DimensionKey`
in downstream code would need a new branch — unlikely, since the vocabulary is a
closed, read-by-tag contract.

## Known limitations

- **Object-pointer input assembles caller-side** — a node references its parent
  object, so you must build top-down (parents before children) and collect the
  nodes into a flat list. This fits an existing tree you can walk; ingesting
  unordered flat/relational rows (a `parent_id` join) would want a nominal-ref
  path, not yet provided.
- **Children order** — by explicit `order` ascending, else alphabetically by name
  (a deterministic default; the V1 authored order is not carried over to V2).
