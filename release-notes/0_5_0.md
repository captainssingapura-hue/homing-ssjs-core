# Release 0.5.0 — The Rigid-Tree Document

0.4.0 made the **workspace** a product that browses a studio's catalogue as an
interactive tree. 0.5.0 makes the **document** a rigid tree too. The tree
substrate generalizes into one normalized, explicitly-leveled shape (RFC 0040);
RFC 0039's renderer lands that shape as a *foldable document view* inside the
workspace — a table of contents that folds the body in sync, with the selected
node highlighted; and RFC 0042 gives authors a strongly-typed **leveled DSL**
for writing one directly — `RigidDoc`, `ComposedDoc`'s successor.

The through-line: the same rigid tree that draws the catalogue now models the
document, and a human can author it in code that reads like the tree.

## What's in the box

### RFC 0040 — the normalized rigid tree (substrate)

One explicitly-leveled normalized tree becomes the single shape the
`TreeRenderer` consumes, and each source encoding contributes one
context-free `normalize(spec)` function.

| Piece | Role |
|------|------|
| `TreeLevel` (`L0…L18`) | A typed, capped level ladder (DomOpsParty's pattern); depth lives in `level()` alone. |
| `NormalizedNode` | One canonical node — typed level + dimensions + children — replacing per-adapter node records. |
| `TreeNormalizer<SPEC>` | The seam: a context-free `normalize` per source, always building a standalone `L0` tree. |
| `graft` / `shift` | A pure recursive level shift; a sub-tree built once attaches, unchanged, at any depth (`> L18` is a hard error). |
| `CatalogueTreeAdapter` | Retrofitted as the first normalizer; now **descends `OfStudio` portals**, so a multi-studio umbrella renders its whole synthetic forest. |

### `homing-studio-starter` — the workspace, out of the box (new module)

`StudioStarterFixtures` brings the Studio Workspace (Navigator + catalogue tree
+ leveled Open) to a downstream studio with **zero** wiring — the batteries-
included entry point above the whole stack.

### RFC 0039 — the rigid-tree document, live

The document model decouples **structure** (a `NormalizedNode` tree, nav-only)
from **content** (a position-keyed provider seam), serialized as one two-part
`{structure, content}` payload and rendered by a new substrate view.

| Piece | Role |
|------|------|
| `ComposedDocNormalizer` | `ComposedDoc` → `DocTree` (structure + content providers) — the first `TreeNormalizer` front-end. |
| `DocTreeJsonWriter` / `/doc-tree` | Serializes the two-part payload; resolves by uuid **or** by leveled tree path. |
| `renderDocTree` (`DocTreeRendererModule`) | TOC = the substrate `TreeRenderer`; selection scrolls **and highlights** the body; expand/collapse **folds the body subtree in sync**; focus-gated arrow-key navigation. |
| `DocTreeWidget` / `DocTreeViewer` | The widget + fake-AppModule host (RFC 0024 pattern); a floating **Export HTML** pill with an *include / exclude page chrome* toggle — parity with `ComposedWidget`. |

**Leveled Open.** Every catalogue leaf now opens through one
`SingleWidgetWorkspace` shell, late-binding the right widget from the node's
kind — `SvgWidget` / `ComposedWidget` / `DocTreeWidget`, or a client-side
`LegacyRedirectWidget` for other kinds. The path *is* the URL (`l0,l1,…`, no
uuid), resolved server-side by `ForestPathResolver` (`/open-content`,
`/open-refs`), so the page URL and the breadcrumb finally share one source of
truth. The legacy server-side `/open` redirect retires. Composed leaves open
through the rigid-tree renderer.

### RFC 0042 — the leveled tree-builder DSL

`RigidDoc` is `ComposedDoc`'s successor: a document authored as a *tree* of
titled sections, so it nests — and therefore folds at every level, not just as a
whole.

```java
RigidDoc.root(uuid, "Title", "summary", "DEMO")
    .l1("Section A")
        .text("lead-in prose")
        .l2("A.1")
            .code("x = 1", "java")
        .l2build()
    .l1build()
    .build();
```

One builder type per level (`L0…L6` shipped; the ladder extends to `L18` by the
same template), so the **compiler** rejects a skipped level, an unclosed node,
or a level past the cap — `.lN` names the level, the indented chain draws the
tree. It is pure sugar over the substrate: structure normalizes to
`NormalizedNode`, a leaf's segments become a `ComposedLeaf` bundle (RFC 0041),
output is a `DocTree`. A worked demo — **RigidDoc — A Leveled Tree-Builder
Demo** — ships in Building Blocks, exercising multi-level nesting, a branch with
a lead-in, every content kind, and a multi-segment leaf.

### RFC 0041 — structure vs content (`ComposedLeaf`)

A node's content is a **bundle** — a flat, ordered `ComposedLeaf` of segments
rendered in the node's body, distinct from the structure that drives the TOC.
The doc-tree content pipeline now carries bundles (a node may hold several
segments, or a lead-in above its children).

## Breaking changes

- **`ContentProvider.content()` now returns a polymorphic `LeafContent`** (a
  sealed type; `ComposedLeaf` — a flat segment bundle — is the first kind)
  rather than a single `Segment`. The `/doc-tree` content map keys each node to
  that payload's wire shape (an **array** of segment objects for a
  `ComposedLeaf`); the serializer dispatches with an exhaustive switch, so a new
  leaf-content kind can't be added without a wire shape being chosen. Internal
  to the RFC 0039 doc-tree path; `ComposedDoc` authoring and the legacy `/doc`
  viewer are unchanged.

## RFCs + demos filed this cycle

- **RFC 0040** — *The Normalized Rigid Tree* (implemented: the catalogue tree
  rides it; the synthetic multi-studio forest lights up).
- **RFC 0041** — *Structure vs Content: ComposedDoc as a Rigid-Tree Leaf*
  (design; `ComposedLeaf` landed as the bundle type).
- **RFC 0042** — *A Leveled Tree-Builder DSL* (design; the DSL + `RigidDoc`
  implemented to `L6`).
- **Demo** — *RigidDoc — A Leveled Tree-Builder Demo* (Building Blocks).

## Known limitations carried forward

- The leveled builder ships `L0…L6`; the remaining rungs to `L18` are a
  mechanical code-gen extension (per RFC 0042's open question).
- RFC 0041 and RFC 0042 are filed as design; `ComposedDoc` is untouched and the
  two coexist — the corpus migrates doc-by-doc, opt-in.
- The legacy flat `ComposedViewer` / `ComposedWidget` remain hosted as a
  parallel fallback during the migration.
