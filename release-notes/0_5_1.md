# Release 0.5.1 — The Document Pane

A point release on top of 0.5.0. 0.5.0 made the document a rigid tree and gave
it a foldable viewer; 0.5.1 brings that viewer **into the workspace as a live
pane** and polishes the leveled-Open experience around it.

The headline: select a doc in the Navigator to preview it cheaply, then *open*
it — Enter or double-click — to read the whole thing rendered in place, beside
the tree, with its table of contents folding in sync. Browsing stays cheap; the
expensive render happens only on intent.

## What's in the box

### The in-workspace Document pane

A new `DocContentWidget` joins the Studio Workspace as a third pickable pane
(📖 Document), alongside the Navigator and Summary. It renders the **opened**
document in place — the same foldable TOC + synced-highlight body the standalone
viewer ships — so a doc reads live next to the tree, no new tab.

The key move is a **two-tier navigation vocabulary** on the tree, so a pane pays
only for what the user means:

| Tier | Trigger | Message | Consumer |
|------|---------|---------|----------|
| **Select** | arrow keys / single click | `NodeSelected` → `NavigateTo` | Summary (cheap re-label) |
| **Open** | **Enter** / **double-click** | `NodeOpened` → `OpenDoc` | Document (expensive render) |

`TreeRenderer` gains an `onActivate` callback (Enter on the selected row, or a
double-click) distinct from the high-frequency `onSelect`; `TreeWidget` publishes
`NodeOpened`; the `NavigatorSecretary` redirects it as `OpenDoc`. The Document
pane reacts to `OpenDoc` **only** — never to browsing — so the fetch + render is
paid on intent, not on every selection move. Each open dissolves the previous
render's DomOpsParty sub-branch and mounts a fresh one (No DOM Destruction), with
a sequence guard dropping a superseded fetch. Kind-gated: `composed` docs
(ComposedDoc + RigidDoc) render in place; other kinds degrade to a hint.

### Refinements to leveled Open

- **Clickable breadcrumbs.** `ForestPathResolver` now returns a `Crumb(text,
  href)` trail and `/open-refs` emits the catalogue browse URL per crumb, so a
  leveled-Open page's breadcrumb links to its ancestor catalogues — link-for-link
  consistent with uuid-open (both use `CatalogueAppHost.urlFor`).
- **HTML export keeps the TOC.** A content-only export (page chrome excluded)
  retains the full two-pane document — TOC + body — and a `data-export-width`
  hint widens the standalone `<main>` so the sidebar no longer crushes the body.

### Internal

- Typed CSS for the new pane: `StudioStyles` gains `st_doc_pane` (scroll
  container) and `st_doc_empty` (muted empty-state), both on theme tokens. The
  Document pane styles entirely through `css.addClass` — **no inline styles** —
  so the conformance raw-CSS scan stays green.

## Breaking changes

None. The Document pane and the `onActivate` tier are purely additive — the
catalogue Navigator and the standalone doc viewer are unchanged.

## Known limitations carried forward

- The Document pane renders `composed` kinds in place; other kinds (svg / plan /
  app) still degrade to a hint pointing at the Summary pane's Open button.
- `SummaryWidget` and `TreeWidget` still style inline — candidates for the same
  typed-class refactor the Document pane now models.
