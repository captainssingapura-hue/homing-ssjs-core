# Release 0.5.2 — Legacy Markdown, Live

A point release on top of 0.5.1. 0.5.1 brought the rigid-tree doc viewer into the
workspace as a live Document pane — for `composed` docs (ComposedDoc + RigidDoc).
0.5.2 extends that pane to the **whole legacy markdown corpus**: a plain `.md`
doc now reads in the workspace, with a real foldable table of contents derived
from its own headings — **no migration to ComposedDoc required**.

The point: the cost of moving the existing `.md` corpus to the new doc model is
high, so the new doc model comes to the corpus instead.

## What's in the box

### Markdown docs render in the workspace — with a TOC

A new `MarkdownDocNormalizer` transforms any markdown-bodied `Doc` into the same
rigid-tree shape ComposedDoc/RigidDoc produce, so it flows through the existing
`/doc-tree` + `DocContentWidget` pipeline unchanged — the third
`TreeNormalizer` doc front-end, after the composed and rigid ones.

| Piece | Role |
|------|------|
| `MarkdownDocNormalizer` | Parses the markdown's **ATX headings** (`#`…`######`) into the `NormalizedNode` structure, so each section is a foldable, navigable TOC node; the prose between a heading and the next becomes that node's body (one `MarkdownSegment`). |
| `/doc-tree` markdown branch | `DocTreeGetAction` now transforms `kind` `doc` / `markdown` docs (alongside ComposedDoc + RigidDoc) rather than 404-ing. |
| `DocContentWidget` gate | The open-in-pane gate relaxes from `composed`-only to `{composed, doc, markdown}`; other kinds (svg / plan / app) still degrade to a hint. |

Structural nesting is honoured even across skipped heading levels (a `###` under
a `##` is a child), fenced code blocks are respected (a `#` inside ``` ``` ``` is a
comment, not a heading), and a leading `# Title` that duplicates the doc title is
absorbed into the root instead of repeated. A doc with **no** headings stays a
single flat node (its whole body as content) — the cheap fallback.

### `MarkdownDoc` — the one-line wrapper

A new concrete `Doc` record:

```java
new MarkdownDoc(uuid, "Title", markdownString);   // kind "doc" — renders in the pane
```

Unlike the (deprecated, interface-only) `InlineDoc`, this is an instantiable
value — wrap any raw markdown in one line. Two cases it serves directly:

- **Legacy / external `.md`** — read the file and wrap it; it reads in the
  Document pane with no `ComposedDoc` ceremony.
- **Agent skills** — surface a `SKILL.md` for reading in the studio by wrapping
  its text.

## Breaking changes

None. The markdown path is purely additive — a new normalizer, a new dispatch
branch, a widened gate, and a new `Doc` record. ComposedDoc / RigidDoc rendering
and the legacy `/doc` (DocReader) viewer are unchanged.

## Known limitations carried forward

- **Flat fallback for unstructured docs** — a markdown doc with no ATX headings
  renders as one node (no in-doc TOC). Add headings, or migrate to a
  ComposedDoc/RigidDoc, for richer structure.
- **ATX headings only** — Setext (underline `===` / `---`) headings are not
  parsed into the tree; they render inline in the body.
- **One segment per section** — a section's body is a single `MarkdownSegment`
  (prose + inline code/images via Marked). Typed segment composition (SVG, tables
  as first-class nodes, nested doc embeds) remains the ComposedDoc/RigidDoc path.
