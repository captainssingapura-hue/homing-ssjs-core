# Release 0.4.0 — The Workspace Becomes a Product

0.3.0 turned the workspace into a substrate — an event-sourced shell that
records, folds, and projects. 0.4.0 ships the first *product* built on that
substrate: the **Studio Workspace**, a reusable, spec-driven workspace that
browses a studio's entire catalogue forest as an interactive tree, shows the
selected node in a detail pane, and opens any leaf to its full document in one
click. No bespoke app — just a `WorkspaceSpec` on the V2 shell, plus the
tree-rendering substrate (`homing-tree-views`) made concrete.

This cycle also hardens that tree substrate (selection and keyboard nav), files
the document model's next direction (RFC 0039), and de-overloads `Doc.contents()`.

## What's in the box

### `homing-studio-workspace` — the Studio Workspace (new module)

A single `WorkspaceSpec` (kind `studio`) on the `homing-workspace-shell`
substrate, reachable at `?app=genericWorkspace&ws_kind=studio`. Pickable
widgets, a navigation Party, and a studio-side URL resolver — no new AppModule.

| Part | Role |
|------|------|
| `CatalogueTreeGetAction` (`/catalogue-tree`) | Serves a studio's catalogue forest as canonical `TreeNode` JSON (via `CatalogueTreeAdapter`) — sub-catalogues + `OfDoc` leaves, each carrying `displayLabel` / `kind` / `summary`. |
| `TreeWidget` (Navigator 🌳) | Renders the tree with `TreeRenderer`; live-follow keyboard nav; publishes the selected node to the navigation Party. |
| `SummaryWidget` (Summary 📄) | Pure Party consumer — draws the selected node as a card (kind badge / title / summary) with an **Open** button. |
| `NavigatorSecretaryModule` | The navigation Party's secretary (RFC 0028 ActorParty) — redirects each selection to its consumers. |
| `OpenDocGetAction` (`/open?id=<uuid>`) | Resolves a doc UUID to its own `url()` and redirects in regular MPA mode — URL resolution stays studio-side, the generic tree substrate stays URL-free. |

Wired downstream via `Fixtures.harnessGetActions()` in both self-studio
(`StudioFixtures`) and the demo (`DemoFixtures`), each rooted at its own
catalogue. The Generic Workspace control panel now hosts multiple workspace
kinds (`ws_kind`) under one registration rather than a single named app.

### `homing-tree-views` — hardened

- **Selection by reference.** The renderer tracks the selected *row entry*, not
  an id-keyed map. Two catalogue branches that share a display name (e.g.
  `RFCs → Architecture` vs `Case Studies → Architecture`) no longer cross-select
  — the collision is dissolved structurally, regardless of id values.
- **Keyboard navigation.** Up/Down live-follow through the visible rows (each
  move fires selection), Right expands the focused branch, Left folds it. The
  host forwards keys only while the workspace is active.

### `ComposedDoc` — `contents()` de-overloaded

`Doc.contents(String, List<String>)` is renamed to
`contentsRootedAt(String, List<String>)`, so `contents()` is a single,
unambiguous no-arg accessor again. The leveled variant is slated for retirement
under RFC 0039.

## Notable RFCs + case studies filed this cycle

- **RFC 0038** — *Intention-First Inline Model*: authorial intent as typed,
  per-domain vocabularies; retires inline emphasis (design).
- **RFC 0039** — *ComposedDoc as a Positional Content-Tree*: decouple structure
  (an owned, leveled node tree on the tree-views substrate) from optional
  polymorphic content, inline the payload, and dissolve cycles by construction
  (design; a future, opt-in, doc-by-doc migration).
- New **Authoring** RFC sub-catalogue.
- **Case study** — *Tree Keyboard Navigation*: how the substrate made the
  feature nearly free.

## Breaking changes

- `Doc.contents(String, List<String>)` → **`contentsRootedAt(String, List<String>)`**.
  Only `DocGetAction` consumed it, and only `ComposedDoc` overrode it; the
  single no-arg `contents()` is unchanged. No known downstream impact.

## Known limitations carried forward

- The Studio Workspace ships the Navigator + Summary widgets; a default
  per-kind 2-pane layout is deferred.
- RFC 0038 and RFC 0039 are filed as design — the intention-first inline model
  and the structure/content-decoupled `ComposedDoc` are future, opt-in
  migrations; existing docs are untouched.
- Floating-modal drag-detach state is still not journaled (carried from 0.3.0).
