# Release 0.5.3 — Typed Blocks, Trees On Demand

A point release on top of 0.5.2. Where 0.5.1/0.5.2 were about *where* a doc
renders (the workspace pane, the legacy `.md` corpus), 0.5.3 is about *what a
doc is made of*. Two moves:

1. **A structured segment vocabulary** — paragraphs and homogeneous lists as
   strict, directly-constructed block segments (no CommonMark round-trip), on
   typed, length-bounded `Line`/`Title` building blocks, under a tightened
   `Segment` hierarchy that puts the "what may go in a RigidDoc node" fence in
   the type system.
2. **A doc can supply its own tree** — a new `DocTreeSource` seam and
   `RigidDoc.fromNode(...)` let a Doc hand the endpoints a pre-built `DocNode`
   tree, a **sibling to the leveled DSL** rather than a replacement.

Plus a real RFC 0039 bug fix: inline svg/table/image segments now load their own
bytes instead of the standalone viewer's HTML page.

## What's in the box

### Typed block segments — paragraphs and lists

Three new `Segment` kinds, built directly (no markdown escape hatch), each with
its own wire `kind` + client renderer:

| Segment | Renders | Notes |
|---------|---------|-------|
| `ParagraphSegment` | one flowing `<p>` | A list of `Line.Plain` lines joined by a single space — where the wrap falls is an authoring detail. `ParagraphSegment.of(text)` word-wraps a prose string at the line cap. |
| `UnorderedListSegment` | `<ul>` | **Homogeneous** — every item the same concrete `Listable` kind (all paragraphs, or all images for a gallery), enforced at construction. |
| `OrderedListSegment` | `<ol>` | Same homogeneity rule, numbered. |

List items are `Listable` and list segments are **not**, so **a list cannot
contain a list** — it won't compile. Nesting is the tree's job (`DocNode`
children), not a list item's. Items dispatch recursively through the renderer's
own `ctx.renderContent`, so a list of images reuses the image renderer; because
items are never lists, that recursion is depth-1.

### `Line` / `Title` — typed, length-bounded text

New value objects in `…composed.text`:

- **`Line`** — a single line, sealed into `Line.Plain` (literal, cap 81 chars on
  the raw string) and `Line.Articulated` (inline-markup-bearing, cap on the
  *effective* visible length; mark-stripping is stubbed to raw length for now, a
  safe lower bound). `Line.optionalPlain(s)` is the one boundary where an
  authoring `String` (DSL, convenience ctors) crosses into the typed line — blank
  becomes `Optional.empty()`.
- **`Title`** — a non-blank heading line, capped tighter (66 chars) because long
  headings read badly.

Captions and titles across the segment types move from raw `String` to these:
`RelationSegment`, `SvgSegment`, `TableSegment`, `ImageSegment`,
`ComposedSegment`, `DocumentaryWidget` now hold `Optional<Line.Plain>` captions
(with String-taking convenience constructors preserved), and `DocNode.title` is
now a `Title`.

### A tightened `Segment` hierarchy

The flat `Segment` sum gains a sub-hierarchy:

```
Segment
 ├─ RigidSegment           ← the subset a RigidDoc node may hold
 │    ├─ Listable          ← the subset a list item may be
 │    │    ├─ Markdown/Text/Code, Relation, Paragraph
 │    │    └─ Svg/Table/Image
 │    └─ Unordered/OrderedListSegment
 ├─ ComposedSegment        ← doc-in-doc graft — flat ComposedDoc only
 └─ DocumentaryWidget      ← embedded widget — flat ComposedDoc only
```

`DocNode.content()` is now typed `List<RigidSegment>`, so the two kinds a rigid
tree excludes by design — `ComposedSegment` (redundant with the tree's own child
nesting) and `DocumentaryWidget` (interactivity belongs to the workspace) —
**will not compile** inside a RigidDoc. The fence is in the type system, not a
runtime check, and is RigidDoc-scoped: a flat `ComposedDoc` keeps the full
`Segment` surface.

### Node captions — a header distinct from the title

A node can now carry an optional **caption**: a highlighted header
(`<h2 class="st_section_title">`) drawn above its body, distinct from its
structural **title** (the TOC heading). `DocNode`, `ComposedLeaf`, and the new
`RigidNodeContent` wrapper all carry `Optional<Line.Plain> caption`.

On the wire this stays backward-compatible: a caption-less node still serialises
as the legacy segment **array**; only a node *with* a caption switches to the
object form `{caption, segments}`. A new client twin, `NodeContent`, normalises
both into a single `{caption, segments}` shape (the front-end mirror of
`ComposedLeaf`), and a `CaptionRenderer` — injected as a `renderCaption` callback
exactly like `renderContent` — owns the `st_section_title` style so the
substrate renderer stays style-free.

### `DocTreeSource` + `RigidDoc.fromNode` — bring your own tree

- **`DocTreeSource`** — a Doc that computes its **own** `DocTree` (e.g. a
  catalogue mirror, possibly lazily at request time) implements this interface;
  `DocTreeGetAction` / `DocTreeContentGetAction` call `toDocTree()` directly,
  taking precedence over the `instanceof`-kind dispatch. This decouples doc-tree
  serving from the concrete doc types.
- **`RigidDoc.fromNode(uuid, summary, category, root)`** — build a `RigidDoc`
  from a pre-assembled `DocNode` tree instead of only through the leveled `Rigid`
  DSL. A caller that already holds an arbitrary tree (transformed from another
  source) assembles the nodes and wraps them here — the "sibling to the DSL" of
  the commit's title.
- **`RigidNodeContent`** — the provider-layer node-content wrapper (the former
  `SimpleListSegment`, demoted from a `Segment` to node content — "a list of
  segments" is a node's content, not a segment), with `of(...)` / `captioned(...)`
  builders.
- **Rigid DSL** gains `.svg(doc[, caption])` and `.image(doc[, caption])` chain
  methods alongside the existing `text`/`markdown`/`code`/`relation`.

### Inline resources load their own bytes — RFC 0039 fix

`GET /doc-tree-content?id=<root>&path=<a/b/c>&seg=<i>` (new
`DocTreeContentGetAction`) serves the **raw bytes** of a resource-backed inline
segment (svg / table / image) embedded in a doc tree. `DocTreeJsonWriter` now
emits this URL for those segments instead of the resource's standalone-viewer
`url()`.

This fixes the RFC 0039 defect where the client renderers fetched the viewer
app's **HTML page** and either executed its `appMain` (the SVG crash) or failed a
`JSON.parse` (the image error). The segment is addressed by the registered root's
uuid plus the same positional path/seg keys the writer already emits, so an
embedded doc needs no separate UUID registration — its addressability is its
position inside the root.

## Breaking changes

Unlike 0.5.1/0.5.2, this release carries **source-level breaking changes** for
callers that construct these types directly. DSL/String-based authoring is
largely shielded by convenience constructors; direct constructors changed:

- **`DocNode`** — canonical constructor is now
  `DocNode(Title, Optional<Line.Plain> caption, List<RigidSegment>, List<DocNode>)`.
  `title` moved from `String` → `Title`; `content` from `List<Segment>` →
  `List<RigidSegment>`. A caption-less convenience constructor
  `DocNode(Title, List<RigidSegment>, List<DocNode>)` is provided.
- **`Rigid.segment(...)`** now takes `RigidSegment`, not `Segment` — a
  `ComposedSegment` / `DocumentaryWidget` no longer compiles into a RigidDoc.
- **Caption fields typed** — `RelationSegment`, `SvgSegment`, `TableSegment`,
  `ImageSegment`, `ComposedSegment`, `DocumentaryWidget` canonical constructors
  take `Optional<Line.Plain>` (was `Optional<String>`); String convenience
  constructors are retained.
- **`ComposedLeaf`** canonical constructor gains an
  `Optional<Line.Plain> caption` first parameter (the `List<Segment>`-only
  constructor still works, defaulting to no caption).
- **`DocTreeJsonWriter.write`** gains a `write(tree, rootId)` overload; the
  single-arg `write(tree)` is retained (unrooted content URLs) for tests and
  callers with no root uuid, but production callers must pass the root id for
  inline resources to resolve.

The doc-tree wire format stays backward-compatible (caption-less nodes are
unchanged), as does the standalone `/doc` (DocReader) viewer.

## Deprecations

- **`MarkdownSegment`** and **`TextSegment`** are now `@Deprecated` (signpost
  only — no removal, still fully functional). The raw-CommonMark / free-form
  prose escape hatch is being superseded by the strict, directly-constructed
  block segments on `Line` (`ParagraphSegment` is the first; a quote successor is
  planned). Prefer the typed building blocks going forward.

## Known limitations carried forward

- **Articulation marks not yet stripped** — `Line.Articulated.effectiveLength()`
  counts the raw length until the inline-mark grammar phases in. It's a safe
  lower bound (nothing over-long slips through), but the visible-length cap isn't
  biting on visible text yet.
- **Lists don't nest** — by design; deeper structure is the rigid tree's job
  (`DocNode` children), not a list item's.
- **Flat `ComposedDoc` list authoring is best-effort** — the supported list path
  is the rigid-tree one; a flat composed doc emitting a list reuses the list's
  own leveled URL for nested resources rather than a precise per-item URL.
